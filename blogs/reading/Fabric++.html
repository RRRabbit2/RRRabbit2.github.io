    # Fabric++

    # Fabric原始的交易流程

    Fabric与现有的数据库系统对比，存在的缺陷为：事物的流水线化。虽然在拜占庭环境中处理部分恶意节点事务时，区块链表现得比传统的分布式数据库系统强，但是在事务处理的其他方面区块链要落后分布式数据库很多年。

    从两个方面改进：交易重排序和提前交易中断

    第二节：从概念的角度讨论了Fabric的交易流

    第三节：并行控制方面的相关工作

    第四节：详细讨论了Fabric流的缺陷，怎样用数据库的方法解决这些问题

    第5.1节：对交易排序进行了提升。默认情况下，由于交易的随机排序，导致了不必要的序列化冲突。提出了一种提前重排序的机制，提高了有效交易的数量。

    第5.2节：在任意流水线的阶段，提前中断交易，让流水线不会被未被最终执行的交易堵塞。对Fabric进行了扩展，形成Fabric++。

    第六节：实验。Smallblank和自定义的工作负载下的测试，交易成功量提高。改变区块大小、通道数量、客户端数量展示可扩展性。延迟比vanilla Fabric低。

    # 相关研究工作

    1. 交易吞吐

      • 分析交易之间的依赖关系，将依赖关系图分成互不相交的子图（不适合，Fabric在模拟交易执行阶段采用了并行执行？？）
      • 从底层对并行控制进行优化（不适合，因为真正影响Fabric交易吞吐量的是顶层的密码学、网络和信任验证）

    2. 交易终止/成功

      • 共享锁（不合适，锁会造成很大的网络开销）
      • 改变交易时间戳，将将交易提交时间变为交易开始的时间（√，进一步改进为重新排序）

    # 消除界线

    感兴趣的方面：

    1. 与分布式系统共享的术语
    2. 潜在的可使用的数据库技术
    # 4.1 事务排序的重要性

    order阶段的序列顺序，对最终真正执行的事务的数量有影响。

    # 4.2 事务的生命周期

    产生冲突的时间：

    • 模拟执行阶段（块间的冲突）

      前一个块执行阶段的事务和后面的块中模拟执行阶段的事务发生冲突（对同一个值进行操作）

    • 排序阶段（块内冲突）

      不同的排序结果会导致最终的执行结果不同（对相同的值进行了操作），发生冲突

    # 5 Fabric++

    # 5.1 重排序的步骤：
    1. 构建冲突图
    2. 计算强连通子图，冲突图拆分（具体使用Tarjan‘s 算法）
    3. 记录每个操作在不同的环中出现的次数
    4. 移除操作，去除环
    5. 对剩下的操作重新排序
    # 5.2 事务提前终止

    1. 模拟执行阶段

      根据读取数据的版本确定是否要提前退出

    2. 排序阶段

      同理

    # 实验

    1. 四个peer节点，一个排序服务器，一个客户端
    2. 自主设计了测试框架（不同于Caliper-只有一个通道，高负载时容易失败，Gauge-缺乏对Fabric的适用性，BlockBench-缺乏适用性）
