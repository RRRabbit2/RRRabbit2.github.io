    # cita的默认出块时间是3秒，可以按照需求配置不同的出块时间。

    出块间隔是有约束的，也需要其他参数联动。 主要是系统吞吐量和响应时间的权衡。 出块间隔长，则系统吞吐量高。因为每个块能收集并打包更多的交易。这时需要调大block limit，否则会因为block limit限制，无法打包所有的交易，导致系统吃不饱。当然这时响应时间就也很长，用户等待交易上链的时间变长。 出块间隔短，则系统吞吐量降低。因为单位时间内用来做共识的次数多了，对应的用来执行交易的时间就少了。这时要调小block limit。如果block limit太大，打包的交易在出块间隔内根本执行不完，系统就被噎住，出块拖延，系统指标开始波动。当然对响应时间是有好处的，不考虑block limit设置不当的问题，出块间隔短就意味着用户等待交易上链的时间短，用户体验会更好。

    其他的因素，比如网络带宽，磁盘IO。其实还好，如果发交易的速度是均衡的，调整出块间隔只是改变了打包的频度，单位时间内消耗的这些资源基本上是不变的。 当然对网络时延是有要求，如果500ms根本就来不及完成共识的两轮投票，那就不行了。不过对于许可链来说，这个应该不会是问题。

    block limit的调整跟节点的硬件配置有关系，但是主要是cpu。如果cpu配置好，那么单位时间内处理的交易就多，block limit就可以调大一点。

    确定了节点硬件配置之后，在正式部署链之前，先做个压力测试，看看block limit设置多大的时候，刚好能保证3s出块，既不拖延，也没有闲置。 然后调整出块间隔，block limit跟着线性联动就可以了。比如出块间隔从3s改成500ms，block limit也缩小到原来的1/6。

    区块链的实时性主要体现在交易上链的速度，也就是出块间隔。CITA 默认出块间隔是 3s，已经远远小于比特币以太坊等区块链系统。而且 CITA 的出块间隔是可以配置的，如果用户的网络情况和机器配置比较好，可以进一步减小出块间隔。 同步确实会对节点的性能产生一定的影响，尤其是高度差比较大。比如其他节点已经运行了很长时间，突然加入一个全新的节点。CITA 提供的方案是快照。可以由其他节点生成一个快照文件，然后直接导入这个新节点，避免了同步操作对网络和其他产点产生影响。

    # cita一个块中可以容纳多少笔交易

    CITA 采用了更为精确的方式来进行对块内计算量的限制，包括计算复杂度，存储大小，系统带宽等等考虑，重新制定了每个交易对链内系统资源的使用规则，使用类似于以太坊的配额管理制度。CITA 采用的是跟以太坊类似的方案。每个块有一个 block quota limit，不是以交易笔数或者大小为限制，而是以处理块中所有交易的计算量为限制。这个 limit 也是可配置的，并且可以动态修改。这个 limit 的设置跟出块间隔有关系。出块间隔小，limit 就要设置小一点，否则当前块处理不完，下一个块就会往后拖，导致实时性变差。

    在sqlcita中，每次交易都是在链上的存证操作，经过验证，每个块目前已知最多可以容纳44笔交易，大约可以容纳的交易数为20笔。

    # 统计代码行数
    1. 最简单的是使用wc -l直接进行代码行数统计。（wc的几个参数的解释：-c 统计字节数；-l 统计行数；-w统计字数）

    • 统计当前目录下的index.php文件的行数

      cat index.php |wc -l
      1

    • 统计web目录下，js文件数量：

      find web/ -name "*.js" |wc -l
      1

    • 统计web目录下所有js文件代码行数

      find web/ -name "*.js" |xargs cat|wc -l 或 wc -l `find web/ -name "*.js"`|tail -n1
      1

    • 统计web目录下所有js文件代码行数，过滤了空行：

      find web/ -name "*.js" |xargs cat|grep -v ^$|wc -l
      1

    • 统计web目录下所有js文件代码行数。过滤注释行：

      find web/ -name "*.js" |xargs cat|grep -v -e ^$ -e ^\s*\/\/.*$|wc -l
      1
    1. 使用find命令统计当前目录下php脚本文件的行数
    find . -name '*.php' | xargs wc -l
    1
